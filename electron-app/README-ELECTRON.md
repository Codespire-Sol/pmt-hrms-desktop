# PMT-HRMS Desktop (Electron) — Packaging & Build Guide

This directory packages the PMT + HRMS stack (Node/Express API + two Vite web
UIs + a bundled Postgres) into a single desktop app:

- **Windows** installer: `PMT-HRMS Setup <version>.exe` (NSIS)
- **macOS** disk image: `PMT-HRMS-<version>-universal.dmg`

Installers are written to `release/`.

> **Golden rule: build each OS on its own OS.** A Windows `.exe` must be built on
> Windows; a macOS universal `.dmg` must be built on macOS. There is no reliable
> cross-compilation because the bundled Postgres binaries and native modules are
> platform-specific. CI uses a `windows-latest` + `macos-latest` matrix for
> exactly this reason.

> **Where the CI workflow lives:** GitHub only runs workflows found at the
> **repository root**, and this `electron-app/` folder is a subdirectory of the
> repo. The active workflow is therefore at the repo root:
> **`<repo-root>/.github/workflows/build-desktop.yml`**. The copy under
> `electron-app/.github/workflows/` is an inert pointer stub and is never run —
> edit the root workflow instead.

> **Self-update:** shipped builds check GitHub Releases on launch and update
> themselves (see "Auto-update" below). The `publish:` block in
> `electron-builder.yml` and the workflow's `--publish always` must point at the
> **same** GitHub repo, or updates won't be found.

## What this directory owns

```
electron-app/
├── electron/                 # runtime (main.js, db.js, server.js)  — owned by app author
├── resources/                # GENERATED by scripts/prebuild.js (git-ignored)
│   ├── api/                  #   api dist + prisma + PROD node_modules
│   ├── pmt-web/              #   pmt-web built dist
│   └── hrms-web/             #   hrms-web built dist
├── build/
│   ├── icon.png              # 1024² square master (generated by make-icons.js)
│   ├── icon.ico              # Windows icon   (generated by make-icons.js)
│   ├── icon.icns             # macOS icon     (generated by make-icons.js)
│   └── entitlements.mac.plist
├── scripts/
│   ├── make-icons.js         # generates build/icon.{png,ico,icns} from the mark
│   ├── prebuild.js           # builds api + web apps, stages resources/
│   └── notarize.js           # afterSign hook, notarizes mac build
├── electron-builder.yml      # packaging config
└── package.json              # (owned by the app author)
```

The app sources live one level up: `../apps/api`, `../apps/pmt-web`,
`../apps/hrms-web`.

## Prerequisites

- **Node.js 22** (CI uses Node 22; the packaged prisma/API tooling needs ≥ 22)
- **pnpm 10** (`npm i -g pnpm`) — the monorepo's package manager
- Windows builds: build on Windows
- macOS builds: build on macOS (Xcode command line tools installed)

Install this directory's dependencies once:

```bash
cd electron-app
pnpm install
```

## Step 0 — Generate app icons

Real square icons are generated from the Codespire mark
(`../apps/pmt-web/public/Favicon.png`, a 508×491 non-square mark):

```bash
node scripts/make-icons.js      # or: pnpm run make-icons
```

It pads/resizes the mark onto a transparent square canvas and writes:

- `build/icon.png` — 1024×1024 square master
- `build/icon.ico` — multi-size Windows icon (16…256, includes the required 256)
- `build/icon.icns` — macOS icon (up to 1024)

Toolchain (all in `devDependencies`, pure/cross-platform):
`sharp` (resize + pad) · `png-to-ico` (.ico) · `png2icons` (.icns). If `.ico` or
`.icns` generation is unavailable on a given machine, the script logs a warning
and still writes `build/icon.png`; electron-builder then derives the missing
format from that PNG, so the build never fails. `electron-builder.yml` points
`win.icon` at `build/icon.ico` and `mac.icon` at `build/icon.icns`.

CI runs this automatically before `prebuild.js`.

## Step 1 — Prebuild (stage the resources)

This compiles the API, generates the Prisma client, builds both web UIs, and
copies everything into `resources/`.

```bash
node scripts/prebuild.js
```

What it does:

1. `../apps/api`: `pnpm install --prod=false` → `pnpm build` (tsc) →
   `pnpm prisma:generate`.
2. `../apps/pmt-web` and `../apps/hrms-web`: `pnpm install` → `pnpm build` (Vite).
3. Copies into `resources/`:
   - `api/dist` + `api/prisma` + `api/package.json` + a **production-only**
     `node_modules` (includes the generated Prisma client) → `resources/api`
   - `pmt-web/dist` → `resources/pmt-web`
   - `hrms-web/dist` → `resources/hrms-web`

> Several steps in `prebuild.js` are annotated with `// TODO(verify)` where the
> exact behavior depends on the api's real scripts (e.g. the Prisma config path
> and generated-client output). Skim those if a build fails.

## Step 2 — Package with electron-builder

Run **one** command for the OS you are on:

```bash
# On Windows:
pnpm exec electron-builder --win --config electron-builder.yml

# On macOS:
pnpm exec electron-builder --mac --config electron-builder.yml
```

Output lands in `release/`.

### Why the API and Postgres are outside the asar

- `electron-builder.yml` ships `resources/{api,pmt-web,hrms-web}` as
  **extraResources**, i.e. as plain files under the app's `resources/` dir — the
  Node backend is spawned as a real child process and cannot run from inside the
  `app.asar` archive.
- `asarUnpack: **/node_modules/@embedded-postgres/**` forces the bundled
  Postgres native binaries out of the asar so they are executable at runtime.

## Signing & notarization

Signing is **optional** for local test builds (the app is ad-hoc/unsigned) but
**required** to distribute without OS warnings.

### Windows (Authenticode)

Set these env vars (or repo secrets in CI):

- `WIN_CSC_LINK` — base64 of your `.pfx`, or a path/URL to it
- `WIN_CSC_KEY_PASSWORD` — the `.pfx` password

electron-builder signs the `.exe` automatically when they are present.

### macOS (Developer ID sign + Apple notarization)

Signing env vars:

- `CSC_LINK` — base64 of your Developer ID `.p12`, or a path/URL
- `CSC_KEY_PASSWORD` — the `.p12` password

Notarization env vars (all three required), consumed by `scripts/notarize.js`
via the `afterSign` hook:

- `APPLE_ID` — your Apple developer account email
- `APPLE_APP_SPECIFIC_PASSWORD` — an app-specific password from appleid.apple.com
- `APPLE_TEAM_ID` — your 10-character Apple Team ID

If any of the three is missing, `scripts/notarize.js` **skips gracefully** so
un-credentialed local builds still complete. The mac config uses
`hardenedRuntime: true` with `build/entitlements.mac.plist` (permits the bundled
local server + child processes).

## Auto-update (self-updating installs)

Packaged builds self-update via **electron-updater** (a `dependency`), wired in
`electron/main.js` (`initAutoUpdate`, called at the end of `boot()` after the
window + tray are up):

- **Guarded to packaged builds only** — `if (!app.isPackaged) return;`, so it is
  a complete no-op in `pnpm start` / dev.
- On launch it calls `autoUpdater.checkForUpdatesAndNotify()`. electron-updater
  reads the **same** `publish:` GitHub config as electron-builder (baked into
  `app-update.yml`) to find newer releases.
- On `update-downloaded` it shows a non-blocking dialog: **Restart now**
  (`autoUpdater.quitAndInstall()`, routed through the app's clean-shutdown path)
  or **Later** (auto-installs on next quit via `autoInstallOnAppQuit`).
- **Never crashes the app:** the whole init is wrapped in `try/catch`, there is
  an `'error'` handler, and the check promise is `.catch()`-guarded — so an
  offline machine, a missing/first release, or a GitHub rate-limit just logs a
  warning and the app keeps running.
- `update-available` / `update-downloaded` / `error` (and progress) are logged.

For updates to be discoverable, the NSIS installer is **per-user**
(`perMachine: false`) so updates apply without an elevation prompt, and
`differentialPackage: true` publishes a blockmap for delta downloads.

> **Requirement:** each release published by CI must land in the GitHub repo
> named in `publish:` (`electron-builder.yml`). The publish repo and the
> updater's lookup are the same block, so they can't drift.

## Step 3 — Automated build & publish on a tag

Pushing a tag that starts with `v` triggers the **repo-root** workflow
`<repo-root>/.github/workflows/build-desktop.yml` (NOT the stub under
`electron-app/.github/`), which:

1. Runs on both `windows-latest` and `macos-latest`.
2. Sets up **Node 22 + pnpm**, then `cd`s into `electron-app` (`working-directory`).
3. `pnpm install`.
4. `node scripts/make-icons.js` — generate the real icons.
5. `node scripts/prebuild.js` — build api + web, stage `resources/`.
6. `pnpm exec electron-builder --config electron-builder.yml --publish always` —
   builds the installer **and** uploads it to a **GitHub Release** (target repo
   from `publish:` in `electron-builder.yml`, currently
   `Codespire-Sol/pmt-hrms-desktop` — a placeholder; confirm the real
   repo name before releasing). It also publishes `latest.yml` /
   `latest-mac.yml`, which the auto-updater consumes.

```bash
# cut a release
git tag v1.0.0
git push origin v1.0.0
```

The release is created automatically and both the `.exe` and `.dmg` are attached.

### Required repo secrets (Settings → Secrets and variables → Actions)

| Secret                        | Purpose                                    | Required?           |
| ----------------------------- | ------------------------------------------ | ------------------- |
| `GITHUB_TOKEN`                | publish the release (auto-provided)        | automatic           |
| `WIN_CSC_LINK`                | Windows code-sign cert (base64/path)       | optional (signing)  |
| `WIN_CSC_KEY_PASSWORD`        | Windows cert password                      | optional (signing)  |
| `CSC_LINK`                    | macOS Developer ID cert (base64/path)      | optional (signing)  |
| `CSC_KEY_PASSWORD`            | macOS cert password                        | optional (signing)  |
| `APPLE_ID`                    | Apple account email                        | optional (notarize) |
| `APPLE_APP_SPECIFIC_PASSWORD` | app-specific password                      | optional (notarize) |
| `APPLE_TEAM_ID`               | Apple Team ID                              | optional (notarize) |

`GH_TOKEN` in the workflow maps to `secrets.GITHUB_TOKEN`; the job has
`contents: write` so it can create the release.

## Pitfalls checklist

- ✅ Build **Windows on Windows**, **macOS on macOS** — never cross-compile.
- ✅ The API and the embedded-postgres binaries live **outside** the asar
  (extraResources + asarUnpack) — otherwise they can't be executed.
- ✅ Run `scripts/prebuild.js` **before** electron-builder — `resources/` must
  exist and be populated.
- ✅ Run `scripts/make-icons.js` before packaging — it (re)generates
  `build/icon.{png,ico,icns}` from the mark. CI does this automatically.
- ✅ The active CI workflow is at the **repo root**, not under `electron-app/`.
- ✅ The `publish:` repo in `electron-builder.yml` must be the real GitHub repo
  (currently the placeholder `pmt-hrms-desktop`) or auto-update + the
  release upload target the wrong repo.
- ✅ For a distributable mac build, supply signing **and** notarization creds;
  otherwise the `.app` will be quarantined on other machines.
- ⚠️ Review the `// TODO(verify)` notes in `scripts/prebuild.js` if the api's
  scripts / Prisma output paths differ from the defaults assumed here.
